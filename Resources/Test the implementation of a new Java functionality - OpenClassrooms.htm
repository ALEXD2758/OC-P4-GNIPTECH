Test the implementation of a new Java functionality

Scenario

Lambazon, the online retailer you have recently started working for, has launched the first version of their web application.

It’s been live for a couple of weeks. This has provided a few more jobs, so they’ve decided to take you on as a full-time employee.

You get to your desk after a meeting, and you see that your team’s group chat has a couple of messages: 

You’re just reading the last comment when you notice an email marked “urgent” has been sent by Chris, the lead developer.

URGENT

Hey!  

Drop whatever projects you’re working on now, because I’ve got a big one for you.  We’ve got to roll out a new module for our web application. It’s going to fit in with the existing web app, except it’s exclusively for internal use. Basically, it should be a user friendly way for our teams to track and manage inventory. It’s actually the same inventory management system project that Lee was working on for a while. Unfortunately, since Lee left, it looks like it’s gotten put on the back burner and no one’s taken ownership of it. We’ve got time crunch on it because the management team is expecting it asap.  Luckily, Lee’s done a lot of work on it, so all you have to do is finish up the last functionality: an interface for creating and validating products before they get entered into the system.

I’ve attached the mock ups for it as well as a PDF copy of Lee’s notes, here.  You can read through them to get yourself familiar with it.  I've glanced through it and looks like Lee's started things out for you with a Create() method in the ProductController class.  Make sure you don’t fiddle around with any of the logic in the existing code; that stuff should be pretty solid, and you won’t need it to write the new functionality. I know a lot of this stuff is new for you, but the notes should give you a good jumping off point.  Also, the Github link is here: https://github.com/OpenClassrooms-Student-Center/P3WellTestedFunctionalityJava. 

Make sure you do a full regime of unit and integration tests; if the management sees any bugs after we roll it out, they’ll let us have it.  Make sure you document your work, too, because our manager is expecting a full presentation once you’re done, and they are a stickler for how you implement code.  

Sorry to dump this on you last minute, but we don’t have anyone else free. You’re a hard worker, though, so I’m sure you’ll be able to figure it out. If you have any questions, don’t hesitate to ask anyone on the team, ok?

Thanks!
Chris

You check out the attachments and click to open the first mock-ups:  “module_create_delete_product” and “product_creation_form”: 

You can tell you’ve got a lot of work ahead of you! You flex your typing fingers and get started.

Deliverables

You will have two types of deliverables to prepare in order to validate the project: the scenario deliverables and the presentation deliverables.

Scenario deliverables: 

Send the deliverables below to your validating mentor 48 hours in advance of your scheduled presentation. This will ensure that the validating mentor will have time to review them in detail.

  • .ZIP with the completed code
  • Link to the entire codebase on GitHub

Be sure to check the Assessment guidelines below in order to ensure that your documents match the required validation criteria.

Failure to send the above deliverables 48 hours in advance will result in the presentation being cancelled. Additionally, the session will be logged as a no-show. If you know you won't be able to make it, notify your mentor as soon as possible so that the presentation can be rescheduled.

Presentation deliverables:

The presentation deliverables below do not need to be uploaded before the presentation, but should be ready to be presented.

  • Mentor presentation (powerpoint or slide deck)

Project Presentation

You'll do an oral presentation of your project with an assessor in order to imitate real-life conditions. Your assessor in this case will play the role of your manager. You will explain your process for developing and testing the new functionality in the inventory system module.

The presentation will be structured as follows:  

  • Presentation and explanation of deliverables (15 minutes)
    • Give a short guided tour of your deliverables.
    • Using a slide deck, explain:
      • How did you implement the functionalities? Why?
      • What tests did you run? Why? How did you implement them? Explain your choices.
      • What constraints or difficulties did you encounter? How did you address them?
      • What would you have changed or done differently?
  • Discussion (10 min)
    • The assessor will ask you questions about your methodology and your deliverables.
    • Be prepared to defend your work, as the validator will challenge your decisions.

The assessor will stop playing the role of the manager 5 minutes before the end of the session so that you can debrief together.

Référentiel d'évaluation

 

Implement Unit Tests for a new functionality in a Java application

  • The tests are congruent with the need and validate the new functionality
  • All tests are evaluated and pass
  • The Java code respects the industry standards (naming conventions, indentation, choice of constructs, documentation)

Implement integration tests for a new functionality in a Java application

  • The tests are congruent with the need and validate the new functionality
  • All tests are evaluated and pass
  • The Java code respects the industry standards (naming conventions, indentation, choice of constructs, documentation)
